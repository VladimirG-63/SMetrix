# Отчет по результатам аудита проекта SMetrix

Проведена полная проверка репозиториев мобильного приложения SMetrix (Android) и серверной части (Spring Boot) на соответствие техническому заданию (Два ТЗ.docx) и RFC-документам (Part1-Part4).

## 1. Что уже реализовано

### Мобильное приложение (Android)
Проект строго следует заявленной архитектуре Clean Architecture + MVVM и принципам Offline-first:
*   **Слой данных (Room DB):** Полностью описаны все Entity (`Project`, `ProjectRoom`, `EstimateItem`, `Worker`, `WorkTask` и др.). Настроены связи `ForeignKey` с каскадным удалением. 
*   **Миграции:** `AppDatabase` содержит миграции (MIGRATION_1_2, MIGRATION_2_3, MIGRATION_3_4). Метод деструктивной миграции (`fallbackToDestructiveMigration`) не используется, что полностью удовлетворяет требованиям продакшена.
*   **Точность расчетов:** Исключено использование `float`/`double`. Все финансовые операции и площади реализованы через `BigDecimal`, которые сохраняются в БД как `String` с помощью `BigDecimalConverter`.
*   **Синхронизация:** Настроен `SyncManager` и `SyncWorker` (WorkManager) для пакетной (batch) передачи данных на сервер. Поддерживается Optimistic Locking (разрешение конфликтов через `version`). Написан слой UI для работы с конфликтами (`ConflictResolutionActivity`).
*   **Безопасность:** Работа с токенами реализована через `EncryptedSharedPreferences` (`SecurePrefsHelper`).

### Серверная часть (Backend / Spring Boot)
*   **Базовая структура:** Развернут стек Spring Boot 3 + PostgreSQL + Spring Data JPA. Настроены все основные сущности и репозитории.
*   **Авторизация (JWT):** Подключен Spring Security. Реализованы endpoints регистрации, логина, обновления токенов (`/auth/register`, `/login`, `/refresh`) в `AuthController`.
*   **Шифрование:** Имеется `SecurityConfig` с `BCryptPasswordEncoder`. Пароли хранятся в БД в захэшированном виде.
*   **Синхронизация:** Полностью реализован `SyncService` (`/sync/push`, `/sync/pull`). Поддерживается пакетная обработка и разрешение конфликтов (сравнение `clientVersion >= serverVersion`).

## 2. Что осталось сделать (Не реализовано)

1.  **Парсер ФГИС ЦС (Критично)**
    *   В серверной части присутствует `FgisParserService` и `FgisScheduler`, однако они являются лишь **заглушками** (Mock). Сервис не делает HTTP-запросов к Минстрою и не использует `Apache POI Streaming API (SXSSF)` для парсинга Excel-файлов, как этого требует ТЗ.
    *   Вместо этого `FgisParserService` сохраняет в базу 4 жестко зашитых (hardcoded) материала (Цемент, Штукатурка, Краска, Грунтовка).
    *   **Действие:** Необходимо реализовать `ResourceDataProvider` (с retry-логикой), скачивание `.xlsx` файла ФГИС ЦС и его потоковый (SAX) парсинг во избежание `OutOfMemoryError`.

2.  **Обработка несоответствия единиц измерения (UNITS_MISMATCH)**
    *   Хотя модель поддерживает флаг `UNITS_MISMATCH`, полная логика таблицы `UnitConversion` и ручное подтверждение коэффициента пользователем на UI требуют дополнительного тестирования.

## 3. Недочеты и уязвимости проекта

При аудите серверной части были выявлены следующие архитектурные проблемы и уязвимости:

### Уязвимости безопасности (Backend)
*   **Отсутствие Rate Limiting (Брутфорс и спам):** В `AuthController.java` методы `/forgot-password` (высылает 6-значный код на почту) и `/reset-password` не имеют ограничений на количество попыток. Злоумышленник может спамить чужие email-ящики и легко подобрать 6-значный код брутфорсом (всего 1 000 000 комбинаций).
*   **Избыточный CORS (CORS Misconfiguration):** В `SecurityConfig.java` установлено `config.setAllowedOriginPatterns(List.of("*"));` и разрешены все заголовки и методы. Для REST API это создает риск, если API в будущем будет использоваться из браузеров. Рекомендуется ограничить домены.

### Архитектурные и технические недочеты
*   **Отсутствие глобального перехватчика ошибок (@ExceptionHandler):** 
    В `AuthController.java` при неверном логине или пароле выбрасывается `RuntimeException("Invalid email or password")`. Так как в Spring Boot нет настроенного `@ControllerAdvice` с обработчиками исключений, клиент получит HTTP 500 (Internal Server Error) со стектрейсом Java вместо положенного HTTP 401 (Unauthorized) или HTTP 400. Это ведет к утечке информации о внутреннем устройстве сервера и некорректной реакции клиента.
*   **Использование UUID v4 вместо UUID v7:** 
    В мобильном приложении генерация UUID реализована в модели, но на сервере новые пользователи и материалы получают стандартный `UUID.randomUUID().toString()` (версия 4). В ТЗ (пункт про БД) явно рекомендован UUID v7 для сохранения временной сортировки и производительности БД при большом числе записей.

## Вывод
Проект обладает очень хорошей архитектурной базой. Все сложные требования (Offline-first, пакетная синхронизация, расчеты в BigDecimal, миграции Room) выполнены на высоком уровне. Основные задачи для перевода проекта в Production: внедрение настоящего парсера ФГИС ЦС, настройка Rate-limit на авторизацию и добавление глобального обработчика исключений на сервере.
