Проект: Мобильное приложение «Smetrix»
1. Описание проекта и Бизнес-логика
Разработка Android-приложения для мелких застройщиков, прорабов и частных бригадиров в России. Приложение автоматизирует расчет строительных смет, учет материалов и расчет заработной платы рабочих. Главная ценность: отказ от сложного корпоративного ПО (1С) в пользу простого мобильного инструмента, работающего оффлайн на объекте (Offline-first) с использованием актуальных государственных сметных цен.
2. Технологический стек (Строгие ограничения)
Проект пишется с упором на понятный, поддерживаемый код без избыточного усложнения архитектуры. Обязателен строгий контроль кодировки (UTF-8) на всех этапах передачи данных для корректного отображения кириллицы.
Mobile (Client): * Язык: Java (Android SDK). Kotlin НЕ использовать. * UI: Классические XML Layouts (НЕ Jetpack Compose). Стандарт Material Design 3 (Primary: #1976D2, Accent: #FF9800, Background: #F5F5F5).
•	Локальная БД: Room (SQLite) — основной источник данных на устройстве при работе в Offline-first режиме.

Локальная БД хранит:
- проекты пользователя,
- комнаты,
- сметы,
- локальные изменения до синхронизации,
- кэш материалов.

Кэш материалов формируется динамически и включает:
- последние использованные материалы,
- популярные материалы региона,
- материалы текущих проектов пользователя.

Все материалы, полученные с сервера, автоматически кэшируются локально для оффлайн-доступа.

Отрицательные значения запрещены для:

- area
- consumption_rate
- prices
- total_payment
- total_price
Для коэффициентов, скидок и наценок отрицательные значения допускаются.
Для сетевого взаимодействия используется Retrofit 2 + OkHttp.

Все операции синхронизации выполняются через REST API (JSON).

• Синхронизация: Приложение обменивается данными с PostgreSQL через REST API (JSON). Приложение построено по принципу Offline-first: локальная база Room является локальным источником данных на устройстве, а сервер используется для резервного хранения и синхронизации между устройствами.

Все сущности создаются на клиенте с использованием UUID v7. В PostgreSQL используется тип uuid. Это позволяет избежать конфликтов первичных ключей при синхронизации данных с нескольких устройств.

Каждая локальная сущность содержит:
- created_at
- updated_at
- version (long)
- sync_state (ENUM)

Поле sync_state принимает значения:
- SYNCED
- PENDING_CREATE
- PENDING_UPDATE
- PENDING_DELETE
- FAILED
- CONFLICT

Поле updated_at обновляется сервером при успешной синхронизации и используется только для отображения времени последнего изменения. Разрешение конфликтов выполняется исключительно через version (optimistic locking). Клиент не участвует в определении актуальности записи по времени.

Для разрешения конфликтов используется optimistic locking. Сервер сравнивает version записи, отправленной клиентом, с текущей version в базе данных.

Если версия устарела, сервер возвращает HTTP 409 CONFLICT. Клиент помечает запись состоянием CONFLICT и уведомляет пользователя о необходимости разрешить конфликт вручную.
При конфликте пользователю отображается экран сравнения локальной и серверной версии данных с возможностью:
- оставить локальную версию,
- принять серверную версию,
- отменить изменения.

Все операции синхронизации должны быть идемпотентными. Повторная отправка одного и того же запроса не должна приводить к созданию дубликатов данных.

Синхронизация выполняется через WorkManager:
- автоматически при наличии сети,
- при запуске приложения,
- периодически в фоне при наличии сети и разрешений системы Android,
- вручную пользователем через кнопку синхронизации.

Все операции синхронизации выполняются пакетно и внутри транзакций. При частичной ошибке batch-синхронизации успешные операции не должны откатываться.
Ошибки фиксируются только для конкретных записей.
Размер batch синхронизации определяется сервером и может ограничиваться для предотвращения перегрузки сети и памяти устройства.
Backend (Server): * Язык и Фреймворк: Java 17+, Spring Boot 3.x.

Отношения сущностей строго иерархичны.

UnitConversion: id (UUID), fgis_unit (String), app_unit (String), conversion_factor (BigDecimal).
•	Пример: fgis_unit="1000 шт", app_unit="шт", conversion_factor=0.001.

• Ограничения связей и удаление данных:
Все связи между сущностями должны быть реализованы через Foreign Key constraints.
Структура связей:
- Project → ProjectRoom
- ProjectRoom → Opening
- ProjectRoom → EstimateItem
- ProjectRoom → WorkTask
Soft delete применяется только к Project через поле deleted_at. При soft delete проекта дочерние сущности не удаляются физически, но становятся недоступны через UI вместе с проектом.
Остальные сущности удаляются физически через cascade delete. Удаление ProjectRoom должно каскадно удалять:
- Opening
- EstimateItem
- WorkTask
4.3.1. Серверный парсер и автоматизация сбора данных (Backend)
•	Механизм сбора: На стороне сервера реализована фоновая задача (Scheduled Task), которая ежеквартально (4 раза в год) инициирует обращение к открытым данным ФГИС ЦС через прямые HTTP-запросы.
•	Обработка сырых файлов: Парсер скачивает государственные массивы данных (Excel), распаковывает их и извлекает целевые атрибуты: Код ресурса, Наименование, Единицу измерения и Сметную цену с привязкой к конкретному Региону РФ.
•	Дифференциальное обновление (Incremental Update): Алгоритм сравнивает скачанные данные с текущими записями в PostgreSQL. Обновляются только те позиции, у которых изменилась цена, или добавляются новые материалы. Это позволяет избежать полного перезаписывания базы.
•	Изоляция данных (Версионность): Работа парсера и обновление серверного справочника никак не влияют на уже созданные пользователями проекты. Новые цены применяются только при добавлении материалов в новые сметы или при ручном запросе пользователя на актуализацию цен в текущем проекте.
5. Алгоритмы расчетов и Математика
5.1. Геометрический движок (Чистая площадь) Расчет производится внутри сущности ProjectRoom.
•	Черновая площадь стен = ((Length + Width) * 2) * Height
•	Правило «Значимых проемов»: Пользователь вручную добавляет проемы (Окна, Двери). Площадь мелких отверстий (розетки, вытяжки) и скрытых зон не вычитается.
•	Площадь проемов = SUM(Opening.width * Opening.height)
•	Чистая площадь стен = Черновая площадь - Площадь проемов. Если поле manual_area_override заполнено пользователем, автоматический расчет площади игнорируется, и все дальнейшие вычисления материалов/зарплат строятся на основе этого ручного значения (для работы в помещениях со сложной геометрией).
5.2. Расчет стоимости материала: Базовая сметная цена (base_price) корректируется с учетом глобальных настроек проекта.
•	Формула цены за единицу: final_price = (base_price * tax_multiplier) * (1 + logistics_markup / 100).
•	Формула итоговой суммы: total_price = final_price * quantity.
Это позволяет пользователю в один клик добавить к смете НДС (20%) или накинуть процент на логистику (включая ЗСР), сохраняя гибкость для конечного заказчика.
Норма расхода: Вводится пользователем вручную. Если расход неизвестен, предусмотрена кнопка «Подсказать расход» (система предлагает усредненные справочные значения ГЭСН с сервера: тонкий, средний, толстый слой). 
Конвертация единиц: Для автоматического приведения цен используется таблица UnitConversion. При импорте данных или расчете сметы система выполняет поиск соответствия единицы измерения ФГИС (unit_measure) и внутренней единицы приложения.
Формула нормализованной цены:
base_price_normalized = base_price_fgis × conversion_factor
Если соответствие в таблице не найдено, система помечает позицию флагом UNITS_MISMATCH и требует ручного подтверждения коэффициента пользователем, предлагая стандартное значение 1.0.

5.3. Расчет заработной платы (WorkTask) * Сдельная: Автоматически подставляется рассчитанная «Чистая площадь» (или пользователь вводит объем вручную) × rate_value (ставка рабочего).
•	Фиксированная: Вводится итоговая сумма за конкретную задачу, объем игнорируется.
6. Бизнес-процессы и UI/UX Логика
6.1. Добавление позиции (Search-as-you-type) * Система не навязывает автоматические шаблоны работ, все позиции формируются вручную.
Приложение поддерживает локальную работу без авторизации.
Синхронизация с сервером и облачный backup становятся доступны после входа в аккаунт.
•	В верхней части интерфейса или рядом с названием проекта отображается индикатор синхронизации:
- синее облако — данные синхронизированы,
- серое облако с точкой — присутствуют локальные изменения,
- красное облако — ошибка синхронизации,
- желтое облако — обнаружен конфликт данных.

При вводе названия материала поиск выполняется по принципу Search-as-you-type.

Для предотвращения избыточных запросов используется debounce 400ms после прекращения ввода текста пользователем.

Система в первую очередь выполняет поиск по локальному кэшу Room для мгновенной работы в оффлайне.

При наличии сети параллельно отправляется запрос к PostgreSQL Full-Text Search для поиска по полному серверному каталогу материалов.

Результаты локального и серверного поиска объединяются.

Все выбранные пользователем материалы автоматически кэшируются локально для последующего оффлайн-доступа.

Локальный кэш формируется динамически:
- последние использованные материалы,
- популярные материалы региона,
- материалы текущих проектов пользователя.
•	Резервный путь (Ручной ввод): Если материал уникален или отсутствует интернет, пользователь вводит параметры (Название, ЕИ, Цена) вручную. Позиция сохраняется в локальной БД Room.
6.2. Контроль снабжения (Жизненный цикл материала): Каждому материалу присваивается цветовой индикатор статуса (см. ниже). В списках материалов реализована поддержка пакетных операций (Batch Actions): через долгое нажатие (Long Tap) пользователь может выбрать несколько позиций и массово изменить им статус.
Статусы:
•	Серый (Нужно купить): Смета просчитана, этап планирования.
•	Желтый (Заказано): Оплачено, ожидается доставка.
•	Зеленый (На объекте): Физически присутствует на стройплощадке.
6.3. Защита данных (Edit Warning) Если пользователь пытается изменить length, width или height в уже заполненной сущности ProjectRoom, отображается блокирующее предупреждение:

"Внимание! Изменение размеров приведет к автоматическому пересчету материалов и заработной платы рабочих. Продолжить?"

После подтверждения пересчет quantity, total_price и total_payment выполняется автоматически в фоновом потоке.

Все пересчеты выполняются внутри транзакции Room (@Transaction) для предотвращения частичного обновления данных.

UI обновляется реактивно через LiveData после завершения пересчета.
7. Интерфейс (Wireframes)
•	Главный экран: RecyclerView со списком проектов пользователя (отображение через CardView). Присутствует Floating Action Button (FAB) оранжевого цвета для добавления объекта.
•	Экран комнаты: Единая форма со сквозным скроллом (ScrollView). Верхний блок — ввод базовых размеров и вычет проемов (реализованы через удаляемые Material Chips). Нижний блок — динамические списки RecyclerView для Сметы материалов и Зарплат.
•	Sticky Bottom Bar: Темно-синяя панель (#0D47A1), постоянно закрепленная внизу экрана внутри Комнаты. В реальном времени отображает суммы: МАТЕРИАЛЫ: X ₽ | ЗАРПЛАТЫ: Y ₽ | ИТОГО ПО КОМНАТЕ: Z ₽. Цифры округляются до целых значений.
•	Корзина: Реализован механизм Soft Delete. Проекты помечаются через поле deleted_at.
Если deleted_at != null — проект считается удаленным и скрывается из основного списка. Окончательное удаление из БД происходит через 30 дней (учитывая длительные циклы согласования смет с заказчиками в строительстве). Очистка старых записей на сервере выполняется по расписанию (Cron).
8. Безопасность и Юридические требования
•	Хранение данных: В соответствии с ФЗ-152 «О персональных данных», серверная база данных должна располагаться на территории РФ.
•	Отказоустойчивость парсера: На сервере реализована система алертов. Если при квартальном обновлении парсер получает ошибку доступа к ФГИС ЦС (код 403 или изменение структуры Excel), администратор получает уведомление в логах/почте для ручной корректировки маппинга.
